# API Error Response Contract

## Summary

This document defines the standardized API error response body used by the backend and consumed by the frontend.

The key rule is simple: the backend owns the user-facing `message`; the frontend normalizes the structure and displays the message inside the UI.

## Response Body

All API errors should use this shape:

```json
{
  "code": "RATE_LIMIT_EXCEEDED",
  "message": "Too many requests. Please try again later.",
  "status": 429,
  "details": null,
  "errors": null,
  "retryAfterSeconds": 60,
  "creditStatus": null,
  "traceId": "00-..."
}
```

## Fields

| Field | Type | Required | Purpose |
|---|---|---:|---|
| `code` | string | Yes | Stable machine-readable error code. |
| `message` | string | Yes | User-facing message generated by the backend. |
| `status` | number | Yes | HTTP status code repeated in the body. |
| `details` | object or null | No | Extra structured details for specific errors. |
| `errors` | object or null | No | Field-level validation errors. |
| `retryAfterSeconds` | number or null | No | Retry hint for rate-limit errors. |
| `creditStatus` | object or null | No | AI credit state when credit limit is reached. |
| `traceId` | string or null | No | Request trace identifier for debugging. |

## Error Codes

| Code | Status | Meaning |
|---|---:|---|
| `VALIDATION_FAILED` | 400 | Request body or model validation failed. |
| `INVALID_REQUEST` | 400 | Request is structurally invalid or violates input rules. |
| `UNAUTHORIZED` | 401 | Authentication is missing, expired, or invalid. |
| `FORBIDDEN` | 403 | User is authenticated but not allowed to perform the action. |
| `EMAIL_NOT_VERIFIED` | 403 | User must verify email before continuing. |
| `NOT_FOUND` | 404 | Requested resource does not exist. |
| `CONFLICT` | 409 | Request conflicts with current domain state. |
| `RATE_LIMIT_EXCEEDED` | 429 | App-level rate limit was exceeded. |
| `AI_CREDIT_LIMIT_EXCEEDED` | 429 | User has no AI credits left for the current day. |
| `GITHUB_NOT_LINKED` | 409 | User has not connected GitHub. |
| `GITHUB_TOKEN_MISSING` | 409 | GitHub provider row exists but token data is incomplete. |
| `GITHUB_TOKEN_INVALID` | 409 | GitHub token expired, was revoked, or returned bad credentials. |
| `GITHUB_RATE_LIMITED` | 429 | GitHub rate limit was reached. |
| `GITHUB_API_ERROR` | 502 | GitHub could not be reached or returned an unexpected API failure. |
| `INTERNAL_SERVER_ERROR` | 500 | Unexpected server error. |

## Validation Error Example

```json
{
  "code": "VALIDATION_FAILED",
  "message": "One or more validation errors occurred.",
  "status": 400,
  "errors": {
    "email": ["The Email field is required."],
    "password": ["The Password field is required."]
  },
  "traceId": "0HNC..."
}
```

## Rate Limit Error Example

```json
{
  "code": "RATE_LIMIT_EXCEEDED",
  "message": "Too many requests. Please try again later.",
  "status": 429,
  "retryAfterSeconds": 30,
  "traceId": "0HNC..."
}
```

The HTTP response may also include:

```txt
Retry-After: 30
```

## AI Credit Error Example

```json
{
  "code": "AI_CREDIT_LIMIT_EXCEEDED",
  "message": "Daily AI credit limit reached.",
  "status": 429,
  "creditStatus": {
    "planCode": "free",
    "dailyCreditLimit": 5,
    "usedCreditsToday": 5,
    "remainingCreditsToday": 0,
    "resetAt": "2026-06-17T00:00:00+00:00"
  },
  "traceId": "0HNC..."
}
```

> [!IMPORTANT]
> `creditStatus` is separate from `status`. The `status` field always means the HTTP status code.

## GitHub Reconnect Error Example

```json
{
  "code": "GITHUB_TOKEN_INVALID",
  "message": "GitHub connection has expired or was revoked. Please reconnect GitHub.",
  "status": 409,
  "traceId": "0HNC..."
}
```

The frontend uses `code` to decide whether to show a connect or reconnect button.

## Frontend Consumption Rules

The frontend should follow these rules:

- Display the backend `message` by default.
- Use `code` for control flow, not for replacing the backend message.
- Use `errors` for field-level validation display when needed.
- Use `retryAfterSeconds` only to append retry timing for rate-limit messages.
- Use `creditStatus` to update AI credit UI.
- Use `traceId` for diagnostics, not normal user copy.

## Summary

The response contract gives the frontend one stable error shape. The backend controls error copy and metadata; the frontend only normalizes and renders it.
