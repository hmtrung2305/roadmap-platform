# How Browsers Work

## Learning outcomes
By the end of this lesson, you should be able to:

- Describe the browser's main responsibilities during navigation and rendering.
- Explain the DOM, CSSOM, render tree, layout, paint, and compositing stages.
- Describe how JavaScript interacts with rendering and user input.
- Identify common causes of slow or unstable rendering.
- Use browser developer tools to inspect page construction and performance.

## The browser as a platform
A browser is more than an HTML viewer. It combines several systems:

- A networking layer that fetches resources.
- Parsers for HTML and CSS.
- A JavaScript engine.
- A rendering engine.
- Storage and caching systems.
- Security boundaries.
- Developer tools.
- User-interface features such as tabs, history, and permissions.

Browser implementations differ internally, but they follow the same web standards closely enough for web applications to behave consistently when the application uses supported features correctly.

## Navigation begins with a request
When a user enters a URL, the browser:

1. Parses the URL.
2. Resolves the host through DNS.
3. Establishes a network connection.
4. Negotiates TLS for HTTPS.
5. Sends an HTTP request.
6. Receives a response.
7. Determines how to process the response from its headers and content.

If the response is HTML, the browser begins constructing the document.

## HTML parsing and the DOM
The browser reads HTML and creates the **Document Object Model**, or DOM. The DOM is an in-memory tree of nodes representing document structure.

Given:

```html
<main>
  <h1>Products</h1>
  <p>Browse the catalog.</p>
</main>
```

The browser creates a tree containing a `main` element with `h1` and `p` children.

JavaScript can read and modify this tree:

```javascript
const heading = document.querySelector("h1");
heading.textContent = "Featured products";
```

The source HTML and the current DOM may differ after scripts modify the document.

## CSS parsing and the CSSOM
The browser parses CSS into the **CSS Object Model**, or CSSOM. The CSSOM represents styles and the rules used to calculate each element's final values.

The browser must account for:

- Selector matching.
- The cascade.
- Inheritance.
- Specificity.
- Source order.
- Media queries.
- User-agent defaults.

Example:

```css
main {
  max-width: 60rem;
  margin-inline: auto;
}

h1 {
  font-size: 2rem;
}
```

External stylesheets can affect when the browser is able to render styled content because the browser needs the relevant CSS to calculate presentation.

## From DOM and CSSOM to pixels
The rendering pipeline is commonly explained through several stages.

### Style calculation
The browser determines which CSS rules apply and computes styles for elements.

### Render tree construction
The browser creates a structure containing the visual elements that need to be rendered. Some DOM nodes, such as elements with `display: none`, do not produce visible boxes.

### Layout
The browser calculates geometry, including each visible box's size and position.

Layout can depend on viewport size, fonts, content, ancestor dimensions, and neighboring elements.

### Paint
The browser records drawing operations for text, backgrounds, borders, shadows, and images.

### Compositing
The browser combines painted layers into the final image shown on screen. Some visual changes can be handled by compositing without recalculating layout or repainting the full page.

The exact implementation is more complex, but this model is useful for performance reasoning.

## Reflow, repaint, and visual stability
A DOM or style change may trigger different amounts of work.

Changing layout-related properties such as width, height, margin, or font size may require style recalculation and layout. Changing visual properties may require paint. Changes to properties such as `transform` or `opacity` can often be handled efficiently by compositing, although excessive layers also have costs.

Avoid assuming that every change is equally expensive. Use performance tools to measure the real result.

Unexpected movement during page loading causes poor visual stability. Common causes include:

- Images without reserved dimensions.
- Ads or embeds inserted above existing content.
- Web fonts changing text size after loading.
- Components that appear after asynchronous data arrives without reserved space.

Reserve expected space where practical and avoid inserting late content that pushes the user's current target away.

## JavaScript execution
JavaScript can:

- Modify the DOM and styles.
- Handle events.
- Start network requests.
- Schedule timers.
- Read browser storage.
- Perform calculations.

Browser JavaScript on a page is commonly coordinated through an **event loop**. Tasks such as user events, timers, and callbacks are scheduled for execution.

Long-running JavaScript blocks the main thread. While it runs, the browser may be unable to respond promptly to input or update the page.

Example of blocking work:

```javascript
const startedAt = performance.now();

while (performance.now() - startedAt < 3000) {
  // Busy work blocks the main thread for about three seconds.
}
```

Production code should divide expensive work, move suitable work to a worker, reduce unnecessary computation, or redesign the operation.

## Script loading behavior
A normal external script can block HTML parsing while it downloads and executes:

```html
<script src="app.js"></script>
```

`defer` allows HTML parsing to continue and runs the script after parsing, while preserving document order among deferred scripts:

```html
<script src="app.js" defer></script>
```

An `async` script runs when it becomes available, so execution order is not guaranteed relative to other asynchronous scripts:

```html
<script src="analytics.js" async></script>
```

Modules are deferred by default in typical browser behavior:

```html
<script type="module" src="app.js"></script>
```

Choose loading behavior based on dependencies and when the script must execute.

## Resource discovery and loading
As the browser parses HTML, it discovers stylesheets, scripts, images, fonts, and other resources. It may preload or prioritize resources based on type and context.

Performance is affected by:

- Number of requests.
- Resource size.
- Compression.
- Cacheability.
- Server response time.
- Loading priority.
- Whether a resource blocks rendering.
- Whether a resource is actually used.

A large unused JavaScript bundle can delay interaction even when it downloads quickly because the browser still needs to parse, compile, and execute it.

## Browser security boundaries
Browsers isolate web content through mechanisms such as:

- The same-origin policy.
- Process and site isolation.
- Permission prompts.
- Sandboxed frames.
- Content security restrictions.
- Certificate validation.

These controls reduce the ability of one site to read another site's protected data or access local device capabilities without permission.

Security restrictions can appear as development errors, but bypassing them blindly is not a valid fix. The application must use an appropriate server, origin, permission, or policy design.

## Using developer tools
Useful panels include:

- **Elements**: inspect the current DOM and computed styles.
- **Network**: inspect resources, requests, caching, and timing.
- **Console**: view errors and run JavaScript.
- **Sources**: debug scripts and inspect loaded files.
- **Performance**: record main-thread work, layout, paint, and interaction timing.
- **Application**: inspect storage, service workers, and caches.

The panel names vary slightly across browsers.
