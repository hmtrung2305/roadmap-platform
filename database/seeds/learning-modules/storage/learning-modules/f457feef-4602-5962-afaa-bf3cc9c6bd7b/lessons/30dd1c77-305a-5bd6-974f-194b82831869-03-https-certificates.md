# HTTPS and Certificates

## Learning outcomes
By the end of this lesson, you should be able to:

- Explain what HTTPS protects and what it does not prove.
- Describe the purpose of TLS certificates and certificate authorities.
- Outline the major stages of a TLS connection.
- Recognize common certificate and mixed-content failures.
- Inspect the certificate and security state of a site.

## HTTP compared with HTTPS
HTTP defines the messages exchanged between a client and server. Plain HTTP does not protect those messages from observation or modification while they travel across the network.

**HTTPS** is HTTP carried over **TLS**, or Transport Layer Security.

TLS provides three important properties:

- **Confidentiality**: traffic is encrypted so intermediaries cannot normally read the application data.
- **Integrity**: changes to protected traffic can be detected.
- **Authentication**: the client can verify that it connected to a server authorized for the requested host name.

HTTPS does not guarantee that a site is honest, safe, accurate, or free from malicious application code. It secures the connection to the identified site.

## Certificates
A TLS certificate binds a public key to one or more identities, usually DNS host names.

A certificate typically includes:

- The subject or covered names.
- The public key.
- The issuer.
- The validity period.
- The signature algorithm.
- Intended key usages.
- Additional extensions and constraints.

Modern certificates normally list covered host names in the **Subject Alternative Name** extension.

A certificate for `example.com` does not automatically cover every unrelated host. A wildcard such as `*.example.com` usually covers one subdomain level, such as `www.example.com`, but not the apex `example.com` unless that name is also included.

## Public and private keys
The server has a **private key** that must remain secret. The certificate contains or refers to the corresponding **public key**.

The private key proves control during the TLS handshake and participates in establishing secure communication. Anyone who obtains it may be able to impersonate the server until the certificate is replaced or otherwise invalidated.

Private keys must not be committed to a repository, copied into frontend code, or shared through normal documentation.

## Certificate authorities and trust
A **certificate authority**, or CA, signs certificates after validating control of the requested identity according to its issuance process.

Browsers and operating systems maintain trust stores containing approved root certificates. A server usually presents:

1. Its leaf certificate.
2. One or more intermediate certificates.

The browser builds a chain from the leaf certificate through intermediates to a trusted root.

A server may have a valid leaf certificate but still fail if it does not present the required intermediate certificate or if the browser cannot build a trusted chain.

## A simplified TLS handshake
For a modern TLS connection, the browser and server perform a process similar to this:

1. The browser proposes supported TLS versions and cryptographic options.
2. The server selects compatible options and presents its certificate chain.
3. The browser validates the chain, host name, validity period, and certificate constraints.
4. Both sides establish shared session keys.
5. Application traffic is encrypted and integrity-protected using those session keys.

The exact handshake depends on protocol version, connection reuse, and configuration. The important point is that certificate validation happens before the browser trusts the HTTPS connection.

## Host-name validation
The browser compares the host in the URL with the names covered by the certificate.

For this URL:

```text
https://api.example.com
```

The certificate must cover `api.example.com` through an exact name or a valid wildcard pattern.

A certificate can be correctly signed and unexpired but still be rejected because it covers a different host name.

## Expiration and renewal
Certificates are valid only during a defined time period. Servers must replace or renew them before expiration.

Automated renewal reduces operational risk, but automation should be monitored. Renewal can succeed while deployment of the new certificate fails, or validation can fail because DNS or routing changed.

A complete renewal process includes:

- Requesting or issuing the certificate.
- Installing it on the active endpoint.
- Reloading the relevant service.
- Verifying the served certificate.
- Alerting before expiration.

## HTTPS termination
TLS may terminate at different infrastructure layers:

```text
Browser -> CDN or load balancer -> application server
```

The public endpoint may handle HTTPS and then forward traffic to the application. The connection from the proxy to the application may also use TLS, depending on the architecture and security requirements.

Developers need to know which layer owns the certificate, redirects HTTP to HTTPS, and provides the original scheme and client information to the application.

## Mixed content
A page loaded through HTTPS should not load active resources through plain HTTP.

Example of an unsafe resource reference:

```html
<script src="http://example.com/app.js"></script>
```

This is **mixed content**. A network attacker could alter the script and compromise the secure page. Browsers block many mixed-content requests and may upgrade or warn about others.

Use HTTPS URLs for scripts, stylesheets, API calls, images, fonts, frames, and other resources.

Protocol-relative URLs such as `//example.com/app.js` are generally unnecessary in modern applications. Use explicit `https://` URLs or same-origin relative paths.

## Redirecting HTTP to HTTPS
A server commonly redirects plain HTTP requests to HTTPS:

```http
HTTP/1.1 308 Permanent Redirect
Location: https://example.com/path
```

A redirect improves normal navigation, but the first HTTP request is still unencrypted. Browsers can apply **HTTP Strict Transport Security**, or HSTS, after receiving the policy over a valid HTTPS connection.

Example:

```http
Strict-Transport-Security: max-age=31536000; includeSubDomains
```

HSTS configuration requires care. Long durations and subdomain coverage can make an incorrect HTTPS deployment difficult to recover from.

## Common HTTPS failures
### Certificate name mismatch
The certificate does not cover the requested host.

### Expired or not-yet-valid certificate
The current time is outside the certificate's validity period. An incorrect system clock can also cause this symptom.

### Untrusted issuer
The browser cannot build a chain to a trusted root.

### Missing intermediate certificate
The server did not provide a required intermediate certificate.

### Mixed content
The secure page requests one or more resources through HTTP.

### TLS version or cipher mismatch
The client and server cannot agree on compatible cryptographic settings.

### Local interception
Security software, corporate proxies, or development tools may insert a locally trusted certificate to inspect traffic. This changes the trust path and should be understood before accepting it.

Do not advise users to ignore browser certificate warnings as a normal solution. Find and correct the cause.

## Inspecting a certificate
In a desktop browser:

1. Open an HTTPS site.
2. Open the site's connection or security information.
3. View the certificate.
4. Record:
   - Covered host names
   - Issuer
   - Valid-from and valid-until dates
   - Certificate chain
5. Confirm that the current host matches a covered name.

You can also inspect response behavior with:

```bash
curl -I https://example.com
```

For deeper TLS inspection, tools such as `openssl s_client` may be available, but their output requires careful interpretation.
